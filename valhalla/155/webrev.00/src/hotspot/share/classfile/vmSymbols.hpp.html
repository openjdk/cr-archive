 684 // Here are all the intrinsics known to the runtime and the CI.
 685 // Each intrinsic consists of a public enum name (like _hashCode),
 686 // followed by a specification of its klass, name, and signature:
 687 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 688 //
 689 // If you add an intrinsic here, you must also define its name
 690 // and signature as members of the VM symbols.  The VM symbols for
 691 // the intrinsic name and signature may be defined above.
 692 //
 693 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 694 // you can also define an intrinsic's name and/or signature locally to the
 695 // intrinsic, if this makes sense.  (It often does make sense.)
 696 //
 697 // For example:
 698 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 699 //     do_name(     foo_name, "foo")
 700 //     do_signature(foo_signature, "()F")
 701 // klass      = vmSymbols::java_lang_Object()
 702 // name       = vmSymbols::foo_name()
 703 // signature  = vmSymbols::foo_signature()
 704 //
 705 // The name and/or signature might be a "well known" symbol
 706 // like "equal" or "()I", in which case there will be no local
 707 // re-definition of the symbol.
 708 //
 709 // The do_class, do_name, and do_signature calls are all used for the
 710 // same purpose:  Define yet another VM symbol.  They could all be merged
 711 // into a common 'do_symbol' call, but it seems useful to record our
 712 // intentions here about kinds of symbols (class vs. name vs. signature).
 713 //
 714 // The F_xx is one of the Flags enum; see below.
 715 //
 716 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 717 //
 718 //
 719 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 720 //
 721 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 722 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 723 // of the replacement code may differ from the semantics of the replaced code.
 724 //
 725 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 726 // treated in some other special way by the compiler. For example, the compiler
 727 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 728 // defined in the StringBuilder and StringBuffer classes, see
 729 // Compile::should_delay_string_inlining() for more details).
 730 //
 731 // Due to the difference between the semantics of an intrinsic method as defined
 732 // in the (Java) source code and the semantics of the method as defined
 733 // by the code in the VM, intrinsic methods must be explicitly marked.
 734 //
 735 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 736 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 737 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 738 // class C are present in the loaded class file and are marked;
 739 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 740 // (3) check for orphan methods in class C (i.e., methods for which the VM
 741 // declares an intrinsic but that are not declared for the loaded class C.
 742 // Check (3) is available only in debug builds.
 743 //
 744 // If a mismatch is detected for a method, the VM behaves differently depending
 745 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 746 // A product build will not replace an unmarked library intrinsic method with
 747 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 748 // methods in a product build. The special treatment of a bytecode intrinsic method
 749 // persists even if the method not marked.
