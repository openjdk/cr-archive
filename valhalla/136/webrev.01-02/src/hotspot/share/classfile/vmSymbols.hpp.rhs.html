 683 // Here are all the intrinsics known to the runtime and the CI.
 684 // Each intrinsic consists of a public enum name (like _hashCode),
 685 // followed by a specification of its klass, name, and signature:
 686 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 687 //
 688 // If you add an intrinsic here, you must also define its name
 689 // and signature as members of the VM symbols.  The VM symbols for
 690 // the intrinsic name and signature may be defined above.
 691 //
 692 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 693 // you can also define an intrinsic's name and/or signature locally to the
 694 // intrinsic, if this makes sense.  (It often does make sense.)
 695 //
 696 // For example:
 697 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 698 //     do_name(     foo_name, "foo")
 699 //     do_signature(foo_signature, "()F")
 700 // klass      = vmSymbols::java_lang_Object()
 701 // name       = vmSymbols::foo_name()
 702 // signature  = vmSymbols::foo_signature()
 703 //
 704 // The name and/or signature might be a "well known" symbol
 705 // like "equal" or "()I", in which case there will be no local
 706 // re-definition of the symbol.
 707 //
 708 // The do_class, do_name, and do_signature calls are all used for the
 709 // same purpose:  Define yet another VM symbol.  They could all be merged
 710 // into a common 'do_symbol' call, but it seems useful to record our
 711 // intentions here about kinds of symbols (class vs. name vs. signature).
 712 //
 713 // The F_xx is one of the Flags enum; see below.
 714 //
 715 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 716 //
 717 //
 718 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 719 //
 720 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 721 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 722 // of the replacement code may differ from the semantics of the replaced code.
 723 //
 724 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 725 // treated in some other special way by the compiler. For example, the compiler
 726 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 727 // defined in the StringBuilder and StringBuffer classes, see
 728 // Compile::should_delay_string_inlining() for more details).
 729 //
 730 // Due to the difference between the semantics of an intrinsic method as defined
 731 // in the (Java) source code and the semantics of the method as defined
 732 // by the code in the VM, intrinsic methods must be explicitly marked.
 733 //
 734 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 735 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 736 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 737 // class C are present in the loaded class file and are marked;
 738 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 739 // (3) check for orphan methods in class C (i.e., methods for which the VM
 740 // declares an intrinsic but that are not declared for the loaded class C.
 741 // Check (3) is available only in debug builds.
 742 //
 743 // If a mismatch is detected for a method, the VM behaves differently depending
 744 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 745 // A product build will not replace an unmarked library intrinsic method with
 746 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 747 // methods in a product build. The special treatment of a bytecode intrinsic method
 748 // persists even if the method not marked.
 749 //
