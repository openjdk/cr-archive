  72 .SH DESCRIPTION
  73 .PP
  74 The \f[CB]jarsigner\f[R] tool has two purposes:
  75 .IP \[bu] 2
  76 To sign Java Archive (JAR) files.
  77 .IP \[bu] 2
  78 To verify the signatures and integrity of signed JAR files.
  79 .PP
  80 The JAR feature enables the packaging of class files, images, sounds,
  81 and other digital data in a single file for faster and easier
  82 distribution.
  83 A tool named \f[CB]jar\f[R] enables developers to produce JAR files.
  84 (Technically, any ZIP file can also be considered a JAR file, although
  85 when created by the \f[CB]jar\f[R] command or processed by the
  86 \f[CB]jarsigner\f[R] command, JAR files also contain a
  87 \f[CB]META\-INF/MANIFEST.MF\f[R] file.)
  88 .PP
  89 A digital signature is a string of bits that is computed from some data
  90 (the data being signed) and the private key of an entity (a person,
  91 company, and so on).
  92 Similar to a handwritten signature, a digital signature has many useful
  93 characteristics:
  94 .IP \[bu] 2
  95 Its authenticity can be verified by a computation that uses the public
  96 key corresponding to the private key used to generate the signature.
  97 .IP \[bu] 2
  98 It can\[aq]t be forged, assuming the private key is kept secret.
  99 .IP \[bu] 2
 100 It is a function of the data signed and thus can\[aq]t be claimed to be
 101 the signature for other data as well.
 102 .IP \[bu] 2
 103 The signed data can\[aq]t be changed.
 104 If the data is changed, then the signature can\[aq]t be verified as
 105 authentic.
 106 .PP
 107 To generate an entity\[aq]s signature for a file, the entity must first
 108 have a public/private key pair associated with it and one or more
 109 certificates that authenticate its public key.
 110 A certificate is a digitally signed statement from one entity that says
 111 that the public key of another entity has a particular value.
 112 .PP
 113 The \f[CB]jarsigner\f[R] command uses key and certificate information from
 114 a keystore to generate digital signatures for JAR files.
 115 A keystore is a database of private keys and their associated X.509
 116 certificate chains that authenticate the corresponding public keys.
 117 The \f[CB]keytool\f[R] command is used to create and administer keystores.
 118 .PP
 119 The \f[CB]jarsigner\f[R] command uses an entity\[aq]s private key to
 120 generate a signature.
 121 The signed JAR file contains, among other things, a copy of the
 122 certificate from the keystore for the public key corresponding to the
 123 private key used to sign the file.
 124 The \f[CB]jarsigner\f[R] command can verify the digital signature of the
 125 signed JAR file using the certificate inside it (in its signature block
 126 file).
 127 .PP
 128 The \f[CB]jarsigner\f[R] command can generate signatures that include a
 129 time stamp that enables a systems or deployer to check whether the JAR
 130 file was signed while the signing certificate was still valid.
 131 .PP
 132 In addition, APIs allow applications to obtain the timestamp
 133 information.
 134 .PP
 135 At this time, the \f[CB]jarsigner\f[R] command can only sign JAR files
 136 created by the \f[CB]jar\f[R] command or zip files.
 137 JAR files are the same as zip files, except they also have a
 138 \f[CB]META\-INF/MANIFEST.MF\f[R] file.
 139 A \f[CB]META\-INF/MANIFEST.MF\f[R] file is created when the
 140 \f[CB]jarsigner\f[R] command signs a zip file.
 141 .PP
 142 The default \f[CB]jarsigner\f[R] command behavior is to sign a JAR or zip
 143 file.
 144 Use the \f[CB]\-verify\f[R] option to verify a signed JAR file.
 145 .PP
 146 The \f[CB]jarsigner\f[R] command also attempts to validate the
 147 signer\[aq]s certificate after signing or verifying.
 148 During validation, it checks the revocation status of each certificate
 149 in the signer\[aq]s certificate chain when the \f[CB]\-revCheck\f[R]
 150 option is specified.
 151 If there is a validation error or any other problem, the command
 152 generates warning messages.
 153 If you specify the \f[CB]\-strict\f[R] option, then the command treats
 154 severe warnings as errors.
 155 See \f[B]Errors and Warnings\f[R].

 192 .SH KEYSTORE IMPLEMENTATION
 193 .PP
 194 The \f[CB]KeyStore\f[R] class provided in the \f[CB]java.security\f[R]
 195 package supplies a number of well\-defined interfaces to access and
 196 modify the information in a keystore.
 197 You can have multiple different concrete implementations, where each
 198 implementation is for a particular type of keystore.
 199 .PP
 200 Currently, there are two command\-line tools that use keystore
 201 implementations (\f[CB]keytool\f[R] and \f[CB]jarsigner\f[R]).
 202 .PP
 203 The default keystore implementation is \f[CB]PKCS12\f[R].
 204 This is a cross platform keystore based on the RSA PKCS12 Personal
 205 Information Exchange Syntax Standard.
 206 This standard is primarily meant for storing or transporting a
 207 user\[aq]s private keys, certificates, and miscellaneous secrets.
 208 There is another built\-in implementation, provided by Oracle.
 209 It implements the keystore as a file with a proprietary keystore type
 210 (format) named \f[CB]JKS\f[R].
 211 It protects each private key with its individual password, and also
 212 protects the integrity of the entire keystore with a (possibly
 213 different) password.
 214 .PP
 215 Keystore implementations are provider\-based, which means the
 216 application interfaces supplied by the \f[CB]KeyStore\f[R] class are
 217 implemented in terms of a Service Provider Interface (SPI).
 218 There is a corresponding abstract \f[CB]KeystoreSpi\f[R] class, also in
 219 the \f[CB]java.security\ package\f[R], that defines the Service Provider
 220 Interface methods that providers must implement.
 221 The term provider refers to a package or a set of packages that supply a
 222 concrete implementation of a subset of services that can be accessed by
 223 the Java Security API.
 224 To provide a keystore implementation, clients must implement a provider
 225 and supply a \f[CB]KeystoreSpi\f[R] subclass implementation, as described
 226 in \f[B]How to Implement a Provider in the Java Cryptography
 227 Architecture\f[R]
 228 [https://www.oracle.com/pls/topic/lookup?ctx=en/java/javase/11/tools&id=JSSEC\-GUID\-2BCFDD85\-D533\-4E6C\-8CE9\-29990DEB0190].
 229 .PP
 230 Applications can choose different types of keystore implementations from
 231 different providers, with the \f[CB]getInstance\f[R] factory method in the
 232 \f[CB]KeyStore\f[R] class.
 233 A keystore type defines the storage and data format of the keystore
 234 information and the algorithms used to protect private keys in the
 235 keystore and the integrity of the keystore itself.
 236 Keystore implementations of different types aren\[aq]t compatible.
 237 .PP
 238 The \f[CB]jarsigner\f[R] commands can read file\-based keystores from any
 239 location that can be specified using a URL.
 240 In addition, these commands can read non\-file\-based keystores such as
 241 those provided by MSCAPI on Windows and PKCS11 on all platforms.
 242 .PP
 243 For the \f[CB]jarsigner\f[R] and \f[CB]keytool\f[R] commands, you can
 244 specify a keystore type at the command line with the
 245 \f[CB]\-storetype\f[R] option.
 246 .PP
 247 If you don\[aq]t explicitly specify a keystore type, then the tools
 248 choose a keystore implementation based on the value of the
 249 \f[CB]keystore.type\f[R] property specified in the security properties
 250 file.
 251 The security properties file is called \f[CB]java.security\f[R], and it
 252 resides in the JDK security properties directory,
 253 \f[CB]java.home/conf/security\f[R].
 254 .PP
 255 Each tool gets the \f[CB]keystore.type\f[R] value and then examines all
 256 the installed providers until it finds one that implements keystores of
 257 that type.
 258 It then uses the keystore implementation from that provider.
 259 .PP
 260 The \f[CB]KeyStore\f[R] class defines a static method named
 261 \f[CB]getDefaultType\f[R] that lets applications retrieve the value of the
 262 \f[CB]keystore.type\f[R] property.
 263 The following line of code creates an instance of the default keystore
 264 type as specified in the \f[CB]keystore.type\f[R] property:
 265 .RS
 266 .PP
 267 \f[CB]KeyStore\ keyStore\ =\ KeyStore.getInstance(KeyStore.getDefaultType());\f[R]
 268 .RE
 269 .PP
 270 The default keystore type is \f[CB]pkcs12\f[R], which is a cross platform
 271 keystore based on the RSA PKCS12 Personal Information Exchange Syntax
 272 Standard.
 273 This is specified by the following line in the security properties file:
 274 .RS
 275 .PP
 276 \f[CB]keystore.type=pkcs12\f[R]
 277 .RE
 278 .PP
 279 Case doesn\[aq]t matter in keystore type designations.
 280 For example, \f[CB]JKS\f[R] is the same as \f[CB]jks\f[R].
 281 .PP
 282 To have the tools utilize a keystore implementation other than the
 283 default, you can change that line to specify a different keystore type.
 284 For example, if you want to use the Oracle\[aq]s \f[CB]jks\f[R] keystore
 285 implementation, then change the line to the following:
 286 .RS
 287 .PP
 288 \f[CB]keystore.type=jks\f[R]

 387 .IP \[bu] 2

 389 .IP \[bu] 2

 400 first 8 characters of the alias name specified on the command line, all
 401 converted to uppercase.
 402 If the alias name has fewer than 8 characters, then the full alias name
 403 is used.
 404 If the alias name contains any characters that aren\[aq]t allowed in a
 405 signature file name, then each such character is converted to an
 406 underscore (_) character in forming the file name.
 407 Valid characters include letters, digits, underscores, and hyphens.
 408 .SH SIGNATURE FILE
 409 .PP
 410 A signature file (\f[CB]\&.SF\f[R] file) looks similar to the manifest
 411 file that is always included in a JAR file when the \f[CB]jarsigner\f[R]
 412 command is used to sign the file.
 413 For each source file included in the JAR file, the \f[CB]\&.SF\f[R] file
 414 has two lines, such as in the manifest file, that list the following:
 415 .IP \[bu] 2
 416 File name
 417 .IP \[bu] 2
 418 Name of the digest algorithm (SHA)
 419 .IP \[bu] 2
 420 SHA digest value
 421 .PP
 422 \f[B]Note:\f[R]
 423 .PP
 424 The name of the digest algorithm (SHA) and the SHA digest value are on
 425 the same line.
 426 .PP
 427 In the manifest file, the SHA digest value for each source file is the
 428 digest (hash) of the binary data in the source file.
 429 In the \f[CB]\&.SF\f[R] file, the digest value for a specified source file
 430 is the hash of the two lines in the manifest file for the source file.
 431 .PP
 432 The signature file, by default, includes a header with a hash of the
 433 whole manifest file.
 434 The header also contains a hash of the manifest header.
 435 The presence of the header enables verification optimization.
 436 See \f[B]JAR File Verification\f[R].

 439 The \f[CB]\&.SF\f[R] file is signed and the signature is placed in the

 444 The file has the extension \f[CB]\&.DSA\f[R], \f[CB]\&.RSA\f[R], or
 445 \f[CB]\&.EC\f[R], depending on the digest algorithm used.

 460 .SH JAR FILE VERIFICATION
 461 .PP
 462 A successful JAR file verification occurs when the signatures are valid,
 463 and none of the files that were in the JAR file when the signatures were
 464 generated have changed since then.
 465 JAR file verification involves the following steps:
 466 .IP "1." 3
 467 Verify the signature of the \f[CB]\&.SF\f[R] file.
 468 .RS 4
 469 .PP
 470 The verification ensures that the signature stored in each signature
 471 block (\f[CB]\&.DSA\f[R]) file was generated using the private key
 472 corresponding to the public key whose certificate (or certificate chain)
 473 also appears in the \f[CB]\&.DSA\f[R] file.
 474 It also ensures that the signature is a valid signature of the
 475 corresponding signature (\f[CB]\&.SF\f[R]) file, and thus the
 476 \f[CB]\&.SF\f[R] file wasn\[aq]t tampered with.
 477 .RE
 478 .IP "2." 3
 479 Verify the digest listed in each entry in the \f[CB]\&.SF\f[R] file with
 480 each corresponding section in the manifest.
 481 .RS 4
 482 .PP
 483 The \f[CB]\&.SF\f[R] file by default includes a header that contains a
 484 hash of the entire manifest file.
 485 When the header is present, the verification can check to see whether or
 486 not the hash in the header matches the hash of the manifest file.
 487 If there is a match, then verification proceeds to the next step.
 488 .PP
 489 If there is no match, then a less optimized verification is required to
 490 ensure that the hash in each source file information section in the
 491 \f[CB]\&.SF\f[R] file equals the hash of its corresponding section in the
 492 manifest file.
 493 See Signature File.
 494 .PP
 495 One reason the hash of the manifest file that is stored in the
 496 \f[CB]\&.SF\f[R] file header might not equal the hash of the current
 497 manifest file is that one or more files were added to the JAR file (with
 498 the \f[CB]jar\f[R] tool) after the signature and \f[CB]\&.SF\f[R] file were
 499 generated.
 500 When the \f[CB]jar\f[R] tool is used to add files, the manifest file is
 501 changed by adding sections to it for the new files, but the
 502 \f[CB]\&.SF\f[R] file isn\[aq]t changed.
 503 A verification is still considered successful when none of the files
 504 that were in the JAR file when the signature was generated have been
 505 changed since then.
 506 This happens when the hashes in the non\-header sections of the
 507 \f[CB]\&.SF\f[R] file equal the hashes of the corresponding sections in
 508 the manifest file.
 509 .RE
 510 .IP "3." 3
 511 Read each file in the JAR file that has an entry in the \f[CB]\&.SF\f[R]
 512 file.
 513 While reading, compute the file\[aq]s digest and compare the result with
 514 the digest for this file in the manifest section.
 515 The digests should be the same or verification fails.
 516 .RS 4
 517 .PP
 518 If any serious verification failures occur during the verification
 519 process, then the process is stopped and a security exception is thrown.
 520 The \f[CB]jarsigner\f[R] command catches and displays the exception.
 521 .RE
 522 .IP "4." 3
 523 Check for disabled algorithm usage.
 524 See \f[B]Supported Algorithms\f[R].
 525 .PP
 526 \f[B]Note:\f[R]
 527 .PP
 528 You should read any addition warnings (or errors if you specified the
 529 \f[CB]\-strict\f[R] option), as well as the content of the certificate (by
 530 specifying the \f[CB]\-verbose\f[R] and \f[CB]\-certs\f[R] options) to
 531 determine if the signature can be trusted.

1078 During the signing or verifying process, the \f[CB]jarsigner\f[R] command
1079 may issue various errors or warnings.
1080 .PP
1081 If there is a failure, the \f[CB]jarsigner\f[R] command exits with code 1.
1082 If there is no failure, but there are one or more severe warnings, the
1083 \f[CB]jarsigner\f[R] command exits with code 0 when the \f[CB]\-strict\f[R]
1084 option is \f[B]not\f[R] specified, or exits with the OR\-value of the
1085 warning codes when the \f[CB]\-strict\f[R] is specified.
1086 If there is only informational warnings or no warning at all, the
1087 command always exits with code 0.
1088 .PP
1089 For example, if a certificate used to sign an entry is expired and has a
1090 KeyUsage extension that doesn\[aq]t allow it to sign a file, the
1091 \f[CB]jarsigner\f[R] command exits with code 12 (=4+8) when the
1092 \f[CB]\-strict\f[R] option is specified.
1093 .PP
1094 \f[B]Note:\f[R] Exit codes are reused because only the values from 0 to
1095 255 are legal on Linux and OS X.
1096 .PP
1097 The following sections describes the names, codes, and descriptions of
1098 the errors and warnings that the \f[CB]jarsigner\f[R] command can issue.

1102 limited to) a command line parsing error, the inability to find a
1103 keypair to sign the JAR file, or the verification of a signed JAR fails.

1239 .SH EXAMPLE OF SIGNING A JAR FILE
1240 .PP
1241 Use the following command to sign \f[CB]bundle.jar\f[R] with the private
1242 key of a user whose keystore alias is \f[CB]jane\f[R] in a keystore named
1243 \f[CB]mystore\f[R] in the \f[CB]working\f[R] directory and name the signed
1244 JAR file \f[CB]sbundle.jar\f[R]:
1245 .RS
1246 .PP
1247 \f[CB]jarsigner\ \-keystore\ /working/mystore\ \-storepass\f[R]
1248 \f[I]keystore_password\f[R] \f[CB]\-keypass\f[R]
1249 \f[I]private_key_password\f[R]
1250 \f[CB]\-signedjar\ sbundle.jar\ bundle.jar\ jane\f[R]
1251 .RE
1252 .PP
1253 There is no \f[CB]\-sigfile\f[R] specified in the previous command so the
1254 generated \f[CB]\&.SF\f[R] and \f[CB]\&.DSA\f[R] files to be placed in the
1255 signed JAR file have default names based on the alias name.
1256 They are named \f[CB]JANE.SF\f[R] and \f[CB]JANE.DSA\f[R].
1257 .PP
1258 If you want to be prompted for the store password and the private key
1259 password, then you could shorten the previous command to the following:
1260 .RS
1261 .PP
1262 \f[CB]jarsigner\ \-keystore\ /working/mystore\ \-signedjar\ sbundle.jar\ bundle.jar\ jane\f[R]
1263 .RE
1264 .PP
1265 If the \f[CB]keystore\f[R] is the default \f[CB]keystore\f[R]
1266 (\f[CB]\&.keystore\f[R] in your home directory), then you don\[aq]t need
1267 to specify a \f[CB]keystore\f[R], as follows:
1268 .RS
1269 .PP
1270 \f[CB]jarsigner\ \-signedjar\ sbundle.jar\ bundle.jar\ jane\f[R]
1271 .RE
1272 .PP
1273 If you want the signed JAR file to overwrite the input JAR file
1274 (\f[CB]bundle.jar\f[R]), then you don\[aq]t need to specify a
1275 \f[CB]\-signedjar\f[R] option, as follows:
1276 .RS
1277 .PP
1278 \f[CB]jarsigner\ bundle.jar\ jane\f[R]
1279 .RE
1280 .SH EXAMPLE OF VERIFYING A SIGNED JAR FILE
1281 .PP
1282 To verify a signed JAR file to ensure that the signature is valid and
1283 the JAR file wasn\[aq]t been tampered with, use a command such as the
1284 following:
1285 .RS
1286 .PP
1287 \f[CB]jarsigner\ \-verify\ ButtonDemo.jar\f[R]
1288 .RE
1289 .PP
1290 When the verification is successful, \f[CB]jar\ verified\f[R] is
1291 displayed.
1292 Otherwise, an error message is displayed.
1293 You can get more information when you use the \f[CB]\-verbose\f[R] option.
1294 A sample use of \f[CB]jarsigner\f[R] with the \f[CB]\-verbose\f[R] option
1295 follows:
1296 .IP
1297 .nf
1298 \f[CB]
1299 jarsigner\ \-verify\ \-verbose\ ButtonDemo.jar
